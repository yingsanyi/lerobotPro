# S2-PI05 Lite: 基于关键文字增强的 PI0.5 方案

这份方案专门针对你现在的要求：

- 不走复杂的 object graph / target index / edge 标注路线
- 不要求数据集额外提供框、轨迹、OCR 标注
- 保留原始 `pi05` 的数据组织方式
- 只在原始 `pi05` 数据基础上，新增一个很轻量的“关键文字”输入
- 让模型在内部学习“看到图像中的文字并利用这些文字做动作决策”

核心目标是：

> 在原来 `pi05` 的 `image + task + state -> action` 基础上，变成 `image + task + keyword_text + state -> action`，其中 `keyword_text` 明确告诉模型这次任务里哪些文字是关键判别线索，例如 `止痛药`、`维生素`。

---

## 1. 最简设计原则

对于任务：

```text
将止痛药放在左边的盒子，将维生素放在右边的盒子
```

你真正想增加的能力不是：

- 显式检测所有物体
- 显式建图
- 显式选择 target index

而是：

- 让模型知道这次任务里需要特别关注哪些文字
- 让模型从图像里学会识别这些文字
- 让模型把“图像中的文字线索”与动作监督自动对齐

所以第一版最合理的方案就是：

1. 保留原 `pi05` 的数据结构和训练方式。
2. 每条任务或每条轨迹新增一个 `keyword_text` 字段。
3. 在 processor 里把 `task` 和 `keyword_text` 拼成统一 prompt。
4. 模型内部增加一个轻量 keyword-aware 模块，让关键字对视觉 token 的注意力更强。
5. 其余目标定位、时序决策、抓取顺序，继续交给 `pi05` 主干自己学。

这条路线的优点是：

- 工程量低
- 数据标注成本低
- 不破坏 `pi05` 预训练权重的可迁移性
- 适合先做出第一版能工作的模型

---

## 2. 数据集如何改

### 2.1 保留原 `pi05` 的原始字段

原来 `pi05` 训练需要的字段继续保留：

- `observation.images.*`
- `observation.state`
- `action`
- `task`

这些都不动。

### 2.2 只新增一个字段

新增：

- `keyword_text`

推荐格式：

```text
止痛药, 维生素
```

也就是把这条任务中真正需要模型额外识别的文字列出来，作为一个字符串，而不是复杂结构。

为什么建议用一个字符串而不是 list：

- 更接近 `pi05` 现有 tokenizer / prompt 处理方式
- 更容易直接拼到 prompt 里
- Parquet schema 更简单
- 不需要你额外做对象级对齐标注

### 2.3 这个字段放在哪

最实用的做法有两种。

方案 A：直接作为逐帧列写进 `data/*.parquet`

- 每一帧都带同一个 `keyword_text`
- 虽然有重复，但最容易接入训练代码

方案 B：作为 episode/task 级元数据保存

- 更省存储
- 但训练加载时需要再额外映射到每一帧

如果你要快速落地，优先选 A。

## 3. 用这个任务举例的数据格式

任务：

```text
将止痛药放在左边的盒子，将维生素放在右边的盒子
```

新增字段只需要：

```text
keyword_text = "止痛药, 维生素"
```

### 3.1 一条 frame 的最小示例

```json
{
  "timestamp": 2.4,
  "frame_index": 24,
  "episode_index": 12,
  "index": 1584,
  "task_index": 0,
  "task": "将止痛药放在左边的盒子，将维生素放在右边的盒子",
  "keyword_text": "止痛药, 维生素",
  "observation.state": [0.11, -0.22, 0.35, 0.04, 0.62, -0.18, 0.09, 0.01],
  "action": [0.02, -0.01, 0.03, 0.00, 0.05, -0.03, 0.01, 0.00],
  "observation.images.front_rgb": "video frame"
}
```

这里没有：

- bbox
- OCR 框
- 文字位置标注
- target index
- graph edge

你只告诉模型：

- 任务是什么
- 哪几个文字是关键
- 图像长什么样
- 示教动作是什么

剩下的事情交给模型自己从行为克隆中学。

## 4. Processor 应该怎么处理

最简单的做法不是单独开一条很复杂的分支，而是直接把 `keyword_text` 拼进 prompt。

### 4.1 推荐 prompt 模板

```text
Task: 将止痛药放在左边的盒子，将维生素放在右边的盒子。
Key text: 止痛药, 维生素.
State: <state tokens>
Action:
```

这比原始 `pi05` 多了一行：

```text
Key text: 止痛药, 维生素.
```

### 4.2 为什么这样就够合理

因为 `pi05` 本来就是 VLM policy：

- 图像进入视觉主干
- 文本 prompt 进入语言主干
- 两者在 backbone 里融合
- 最后输出动作

你现在做的只是把“真正关键的文字约束”明确告诉模型，减少它只靠模糊任务语句去猜的难度。

也就是说，这个方案本质上是：

> 用更明确的语言条件，逼模型在视觉里寻找与这些关键字对应的证据。

---

## 5. 模型内部怎么改，才符合你的目标

如果你只做最小改动，实际上可以分两层。

### 5.1 第一层：几乎不改模型主体

只改 processor：

- 读取 `task`
- 读取 `keyword_text`
- 拼接成更强条件的 prompt
- 其余完全沿用 `pi05`

这种方式的优点是：

- 可以最大化复用 `pi05` 预训练权重
- 改动极小
- 很适合做第一版基线

这种方式已经可能有效，前提是你的数据真的迫使模型去读文字。

### 5.2 第二层：增加一个轻量 keyword-aware 模块

如果你想比“只拼 prompt”再强一点，但仍保持简单，推荐只加下面这个内部模块：

1. 从 `keyword_text` token 中取出 keyword token embeddings。
2. 在视觉 token 上做一层 keyword-to-vision cross attention。
3. 得到一个 keyword-aware visual representation。
4. 再把这个表示送回原来的 `pi05` action backbone。

也就是：

```text
image tokens
+ keyword tokens
-> lightweight cross attention
-> enhanced visual-language tokens
-> original pi05 policy head
```

这一层的作用是：

- 让“止痛药”“维生素”这样的关键词在视觉侧更有引导性
- 但不需要你人工告诉模型文字在哪个框里

这就比较符合你说的：

> 其余事情都交给模型内部自动处理。

## 6. 训练目标怎么定

### 6.1 最简单版本

直接用原始 `pi05` 的 imitation / policy loss：

```text
Loss = L_policy
```

这里不需要额外 OCR loss，也不需要额外 graph loss。

如果你的数据设计得好，这个版本就可以先试。

### 6.2 稍微增强但仍然很轻的版本

可以加一个很弱的关键字对齐 loss：

```text
Loss = L_policy + λ * L_keyword_align
```

其中 `L_keyword_align` 不要求框标注，可以做成弱监督：

- 正样本：当前任务里的 keyword_text
- 负样本：其他任务里的 keyword_text
- 目标：让当前图像更偏向匹配当前关键字，而不是别的关键字

但这不是第一版必需项。

我的建议是：

1. 先只跑 `L_policy`
2. 确认关键字 prompt 本身是否有效
3. 再决定要不要上 `L_keyword_align`

## 7. 这个任务应该怎么采数据

任务仍然是：

```text
将止痛药放在左边的盒子，将维生素放在右边的盒子
```

### 7.1 你真正要保证的数据条件

不是多标注，而是要保证这些事情：

- 镜头里能清楚看到包装上的关键字
- “止痛药”和“维生素”在外观上尽量相似，避免模型只看颜色或形状
- 左右位置不能固定，否则模型会学位置 shortcut
- 盒子位置、药品位置、拍摄角度、光照都要变化
- 示教里必须有足够多的“只有读字才能区分”的场景

### 7.2 最关键的采集原则

如果你想让模型真的学会通过文字提高 VLA 能力，就必须让“文字”成为完成任务的必要条件。

具体说：

- 有时止痛药在左，维生素在右
- 有时止痛药在右，维生素在左
- 有时两个药盒颜色很像
- 有时两个药盒大小和形状也很像
- 有时盒子本身标签也会变化

这样模型才不能依赖：

- 固定位置
- 固定颜色
- 固定外形

而会被迫去利用：

- 图像中的文字
- prompt 中的 keyword_text

### 7.3 推荐的数据采集流程

1. 先按原始 `pi05` 的方式录制演示。
2. 每条 episode 记录原始任务句子。
3. 额外写入该任务的 `keyword_text`。
4. 采集时持续打乱止痛药和维生素的位置。
5. 采集一部分 hard negative 数据：外观接近、文字不同。
6. 保证包装正面或主要文字区域在关键帧里尽量清晰可见。

### 7.4 这条任务的推荐标注

对这个例子，不需要每一帧标注对象框。

只需要给整条 episode 或整段轨迹写：

```text
task = 将止痛药放在左边的盒子，将维生素放在右边的盒子
keyword_text = 止痛药, 维生素
```

如果你后面扩展到别的任务，比如：

```text
将阿司匹林放进红盒子，将感冒药放进蓝盒子
```

对应就是：

```text
keyword_text = 阿司匹林, 感冒药
```

## 8. 这个简化方案什么时候会失败

这个方案虽然简单，但有边界。

### 8.1 数据不够“逼着模型读字”时

如果数据里：

- 止痛药总是在左边
- 维生素总是在右边
- 包装颜色永远固定

那模型会学 shortcut，而不是学读字。

### 8.2 图像里文字太小或太模糊时

如果相机分辨率、距离、运动模糊导致文字几乎不可读，那么单靠加 `keyword_text` 没意义。

因为模型没有可用视觉证据。

### 8.3 任务只靠全局 prompt 不足以 disambiguate 时

比如场景里同时有多个都写着“止痛药”的盒子，而你还要求选其中一个特定实例。

这种情况下，只靠 `keyword_text` 可能不够，就需要更细的结构化设计。但这已经超出你现在这个“轻量增强版”的目标了。

---

## 9. 我的建议：你应该怎么落地

如果你现在想快速推进，最合理的路线是：

### Phase 1

只做下面两件事：

- 数据集新增 `keyword_text`
- processor 把 `task + keyword_text` 拼进 prompt

这时候模型主干基本不动，先看效果有没有明显提升。

### Phase 2

如果 Phase 1 有提升，再加一个轻量 keyword-aware cross attention 模块。

仍然不要引入：

- bbox 标注
- graph 标注
- target index 标注

### Phase 3

只有在你发现“仅靠关键字仍然不够”时，再考虑更重的对象级结构。

---

## 10. 结论

你这个判断是对的：

> 第一版不应该把系统做得太重。

对于“将止痛药放在左边的盒子，将维生素放在右边的盒子”这种任务，最务实的方案就是：

- 保留原始 `pi05` 数据格式
- 只新增一个 `keyword_text`
- 把关键字拼进 prompt
- 可选增加一层轻量 keyword-aware attention
- 其余目标定位、阶段切换、动作生成都交给模型内部自己学

这比 graph / target index / OCR 框路线更适合你现在的目标，也更容易先做出有效结果。
